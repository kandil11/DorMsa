# Running & Testing Guide: DorMsa_COMP (Clean Assignment Submission)

This guide provides step-by-step instructions to set up, compile, and run the complete multi-language system from scratch in the **`DorMsa_COMP`** directory. It also includes copy-pasteable test cases to verify the **Inversion of Control (IoC)**, **Dependency Injection (DI)**, and **OTP Verification** services.

---

## 🚀 How to Run the Whole Project from Scratch

Since this folder has been packaged cleanly for assignment submission, `node_modules` dependencies are excluded to minimize file size. Follow these steps to restore dependencies and start the services.

You will need three terminal tabs/sessions:

### Terminal 1: Compile & Run the Java Microservice (Port 5002)
Navigate to the root of the assignment folder `DorMsa_COMP` and boot the Java component assembly server:
```bash
# 1. Navigate to the root directory
cd "/Users/7amada/3rd Year/Semester 6/COMP/DorMsa_COMP"

# 2. Compile the Java root project source code
find src -name "*.java" > sources.txt && javac -d bin @sources.txt && rm sources.txt

# 3. Run the Java application
java -cp bin Main
```

*Expected Log Output:*
```text
=================================================
🚀 Starting DorMsa Java Support Service
=================================================
⚙️ [IoC] Loaded component assembly configuration from components.config
🔍 [IoC Debug] Scanning package: gateway (path: gateway)
...
✅ [IoC] Bootstrap completed successfully. Managed beans: 6
🌐 [HTTP Server] REST API is running on port 5002
=================================================
```

## 🖥️ Interactive Web Dashboard & OTP Simulator

The Java microservice serves a beautiful, real-time interactive web dashboard:
* **Dashboard URL**: **[http://localhost:5002/dashboard](http://localhost:5002/dashboard)**

### What you can do on the Dashboard:
1. **Live Notification Feed**: View all processed emails, SMS, and ticket routings generated by your platform in real-time.
2. **OTP Verification Terminal**:
   * When you register a new user in the React frontend, the OTP code is forwarded to the Java dashboard.
   * You will see the SMS code log on the feed.
   * You can enter the user's phone number and the 6-digit OTP code directly in the **OTP Verification Terminal** widget.
   * Click **Verify Code** to test and verify if the entered OTP is correct or incorrect, getting an instant green success check or red error validation badge.
   * **Tip**: We have implemented **Phone Number Normalization** in Java. You can verify using either E.164 formats (`+2010...`) or raw inputs (`010...`) and they will match correctly!

## 🧪 Test Cases to Verify Java Integration

Open a new terminal window to execute these copy-pasteable `curl` commands and observe the logs.

### Test Case 1: Direct Java Support Routing Test (Port 5002)
Verify the custom DI container routes tickets by keywords, alerts the user, and uses the correct `components.config` email gateway.

* **Run this command**:
```bash
curl -X POST -H "Content-Type: application/json" -d '{
  "id": "java_direct_ticket_101",
  "subject": "Billing dispute - double charge",
  "description": "I paid my deposit but my credit card statement shows two charges.",
  "category": "billing",
  "user": {
    "id": "u_student_999",
    "name": "Mostafa Aly",
    "email": "mostafa@student.msa.edu.eg"
  }
}' http://localhost:5002/api/support/route
```

* **Expected HTTP Response**:
```json
{"status":"success","ticketId":"java_direct_ticket_101","assignedAgent":"Sarah Kandil (Finance Department)","routingNotes":"Automatically routed to Finance because of financial terms detected.","priority":"high"}
```
* **Verify in Java Console**:
You should see:
1. `🔧 [Java Routing Service] Routing ticket: Billing dispute - double charge`
2. `🎯 [Java Routing Service] Assigned to: Sarah Kandil (Finance Department)`
3. `📧 [Java SMTP Email Gateway Component] ───────────` (indicating the configured SMTP component successfully processed the alert).

### Test Case 3: End-to-End System Test (Through Node.js Backend - Port 5001)
Verify that submitting a ticket to the Express server triggers a JWT validation check, routes through the Java microservice, and persists the Java routing details in MongoDB.

1. **Get authorization JWT token by logging in as the seed admin**:
```bash
TOKEN=$(curl -s -X POST -H "Content-Type: application/json" -d '{
  "phone": "K",
  "password": "123456789"
}' http://localhost:5001/api/auth/login | grep -o '"token":"[^"]*' | grep -o '[^"]*$')

echo "Acquired JWT: $TOKEN"
```
*(Alternatively, log in via the React frontend to acquire the authorization header automatically.)*

2. **Submit a ticket to Express utilizing the token**:
```bash
curl -X POST -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" -d '{
  "subject": "System login error",
  "description": "I keep getting an error message when logging in on the main page. Server issue.",
  "category": "technical",
  "priority": "medium"
}' http://localhost:5001/api/support/tickets
```

* **Expected HTTP Response**:
```json
{
  "message": "Support ticket submitted successfully and processed by our Java Routing Engine.",
  "ticket": {
    "subject": "System login error",
    "status": "open",
    "priority": "medium",
    "category": "technical",
    "assignedAgent": "Mostafa IT (Tech Support Lead)",
    "routingNotes": "Routed to Technical department based on keywords."
  }
}
```
* **Verify**:
The Express server correctly delegated calculation rules to the Java service. The ticket created inside MongoDB now contains the Java-assigned agent and notes, proving end-to-end integration!
